AI 업무 자동화 에이전트 보안·승인·감사 정책 설계서

본 문서는 AI 업무 자동화 에이전트를 조직(공공/기업) 환경에 안전하게
도입·운영하기 위해 필요한 보안(Security), 승인(Approval), 감사(Audit)
정책을 정의한다. 정책은 기본적으로 "기본 차단(Default Deny) + 명시적
승인(Explicit Allow)" 원칙을 따른다.

1.  목표 및 적용 범위

-   적용 대상: 문서/메일/공지, 회의록, 일정/태스크, 파일 공유, 데이터
    집계/보고, 외부 리서치 요약
-   적용 계층: Intake(분류), Policy Injection(규칙 주입), Tool
    Execution(행동), Approval Gate(승인), Audit Log(감사)
-   운영 환경: 단일 조직(부서) MVP → 전사 확장 가능

2.  보안 설계 원칙

2.1 기본 원칙 - 기본 차단(Default Deny): 위험 가능성이 있는 행동(대외
발신, 공개 공유, 권한 변경 등)은 기본적으로 차단하고 승인 후 허용 - 최소
권한(Least Privilege): 에이전트/툴은 업무 수행에 필요한 최소 권한만
보유 - 책임 추적(Non-repudiation): 주요 행동은 누가/언제/무엇을 근거로
수행했는지 추적 가능해야 함 - 데이터 최소화(Data Minimization): 필요한
데이터만 처리하고, 민감 데이터는 마스킹/비식별화 우선 - 분리(Separation
of Duties): 작성(생성)과 승인(발신/공유/등록)은 분리하여 사고를 예방

2.2 위험 등급 분류 에이전트가 수행하는 행동과 대상 데이터를 위험
등급으로 분류하여 정책을 차등 적용한다. - R0(낮음): 내부 초안 작성, 내부
문서 템플릿 적용, 비민감 데이터 요약 - R1(중간): 일정 등록 초안, 내부
공유 링크 생성(조직 내 제한), 데이터 집계 결과 생성 - R2(높음): 대외
메일/공지 초안, 외부 링크 공유, 권한 변경, 민감
데이터(개인정보/계약/예산) 포함 문서 - R3(최고): 자동 발신/자동
게시/대량 변경/삭제, 운영 권한 변경, 외부 업로드(공개)

3.  승인(Approval) 정책

3.1 승인 대상(필수) - 외부 도메인 이메일 발송/회신(대외 발신) - 캘린더
일정 생성/수정(조직 캘린더 포함) - 파일 공유(링크 공개, 권한 확장, 외부
공유) - 민감 정보 포함 문서 생성/첨부/공유 - 결재/공문 발송에 준하는
산출물 확정(기관별 정의) - 대량 작업(다수 수신자 발송, 다수 파일
이동/삭제 등)

3.2 승인 단계 모델 - 단일 승인: 1명의 승인자 승인 후 수행 - 다중 승인:
2인 이상 승인(예: 작성자 외 팀장 승인) - 조건부 승인: 특정 조건(수신자
도메인, 등급, 파일 경로)에 따라 자동/수동 승인 전환

3.3 승인 요청(Ask) UI/메시지 표준 승인 요청 메시지는 반드시 아래 정보를
포함한다. 1) 수행하려는 행동(예: 외부 메일 발송) 2) 대상(수신자/공유
대상/캘린더) 3) 영향 범위(외부 노출 범위, 권한 변화) 4) 근거(요청 원문
요약, 관련 문서 링크) 5) 리스크 요약(개인정보 포함 여부, 계약/예산 포함
여부) 6) 대안(내부 검토 후 발송, 제한 공유로 변경 등)

3.4 승인 결과 처리 - 승인: 수행 및 감사 로그 기록 - 반려: 반려 사유 기록
후 REVIEW 상태로 복귀, 수정 루프 재개 - 보류: 일정 시간 초과 시 자동
만료(정책 값으로 설정)

4.  감사(Audit) 및 로깅 정책

4.1 감사 로그 필수 항목 - Request ID / Task ID - 사용자(요청자) /
승인자 - 요청 원문 해시(원문 전체를 저장하지 않는 옵션 포함) - 적용된
규칙 세트 버전(.rules.json 버전, 템플릿 버전) - 수행한 툴/명령, 입력
파라미터(민감 정보 마스킹) - 산출물 메타데이터(파일명, 경로, 링크 권한
수준) - 승인 이벤트(요청 시각, 승인/반려, 사유) - 결과(성공/실패) 및
오류 로그 요약 - 타임라인(상태 전이 기록)

4.2 로그 보존 및 접근 통제 - 보존 기간: 조직 정책에 따름(예: 1년/3년) -
접근 통제: 감사 로그는 보안 담당/관리자 역할만 조회 - 마스킹:
이메일/전화/주민번호 등 개인정보는 저장 전 마스킹 - 무결성: 로그 위변조
방지(서명/해시 체인/Write-Once 스토리지 중 선택)

4.3 산출물 버전관리 - 모든 산출물은 버전(Revision)과 생성 시각을
가진다. - 승인된 산출물만 "공식 버전"으로 표시 - diff(변경점) 요약을
함께 저장(민감 내용은 요약만)

5.  데이터/개인정보(PII) 처리 정책

5.1 개인정보 탐지 - 패턴 기반 탐지(이메일, 전화번호, 주민번호 등) -
키워드 기반 탐지("주민등록", "계좌번호", "인사" 등) - 필요 시 조직 정의
민감 키워드 사전 적용

5.2 마스킹/비식별화 - 기본: 산출물 내 PII 마스킹 후 제공(예: 홍*동,
010-*\*\*\*-1234) - 예외: 업무상 반드시 필요한 경우에만 승인 후 원문
사용 - 외부 발신 시: 원칙적으로 PII 포함 금지(정책으로 강제)

5.3 데이터 이동 제한 - 외부 리서치/웹 검색 툴에 민감 정보 전달 금지 -
외부 API 호출이 필요한 경우 프록시/필터링 계층을 둔다. - 데이터 반출
로그를 별도로 기록(누가 어떤 데이터 범주를 외부로 보냈는지)

6.  툴/명령 실행 통제(Guardrails)

6.1 Allowlist/Blocklist - 명령 allowlist: 허용된 명령만 실행(조직 환경에
맞게 정의) - 위험 명령 blocklist: 삭제/포맷/권한변경/대량작업은 차단
또는 승인 필요

6.2 경로 기반 통제 - 민감 경로(예: 보안 설정, 인사/예산/계약 폴더)는
기본 차단 - 읽기/쓰기 권한을 구분하여 정책 적용

6.3 출력 후처리 - 장문 로그는 절단(truncate) 및 핵심 요약만 제공 - 민감
값(토큰/키/비밀번호) 자동 마스킹

7.  운영 정책

7.1 역할 기반 접근 제어(RBAC) - Requester(요청자): 초안 생성 요청 -
Approver(승인자): 대외/공유/등록 승인 - Auditor(감사자): 로그 조회 -
Admin(관리자): 규칙/템플릿/툴 정책 관리

7.2 규칙/템플릿 변경 관리 - 규칙 변경은 버전 관리 및 변경 이력 기록 -
변경 후 회귀 검증(샘플 시나리오) 수행

7.3 장애/사고 대응 - 오발신/오공유 발생 시 즉시 차단(토글) 가능한 Kill
Switch 제공 - 사고 보고 템플릿 제공(원인/영향/재발방지/조치)

8.  정책 매핑 예시(업무 유형별)

9.  구현 체크리스트(MVP)

□ 요청 분류 및 리스크 태깅 □ rules/템플릿 주입(system.transform) □ 승인
요청 표준(ask 메시지 구성) □ 승인 이벤트 로깅(approve/reject/timeout) □
PII 탐지/마스킹 □ 툴/명령 allowlist+blocklist □ 감사 로그 스키마 및 보존
정책 □ Kill Switch(긴급 차단) □ 샘플 시나리오 회귀
검증(회의→메일/일정/공유)

10. 요약

업무 자동화 에이전트는 생산성뿐 아니라 조직 리스크(오발신, 개인정보
유출, 무단 공유)를 함께 다룬다. 따라서 보안·승인·감사 정책은 기능이
아니라 제품의 핵심이며, 훅/규칙/승인 게이트를 통해 강제되어야 한다.
